Wardress ships a conversational AI agent that helps operators analyze incidents and manage the system. It is an interface *over* the platform: every tool the agent can call runs the same domain logic as the REST routers, so RBAC, SSRF protection, audit logging, and detection semantics stay identical across surfaces. Nothing the model emits is trusted — role and confirmation gating happen in the dispatcher, not in the declarations the model sees.

<Info>
  Source: `backend/app/agent/` (`engine.py`, `tools.py`, `guard.py`, `context.py`). The model is Gemini (configurable in Settings), with optional offline Ollama.
</Info>

## Operating surfaces

You can talk to the agent two ways, and conversation history is stored in the database and shared across both:

1. **Web dashboard** — the chat surface in the React frontend (`agent-web`).
2. **Telegram bot** — an opt-in two-way surface (`agent-telegram`), enabled via the `telegram` Compose profile and configured in Settings.

## Tools and tiers

Each tool declares a **tier** and a **minimum role**. Tiers 0–1 auto-execute; tier 2 and above are frozen and require explicit confirmation.

| Tier | Behavior | Tools |
| :--- | :--- | :--- |
| **0 · read** | Auto-execute (viewer+) | `list_sites`, `get_site`, `get_status_overview`, `list_scans`, `get_scan_findings`, `explain_incident`, `list_alerts` |
| **1 · safe write** | Auto-execute (analyst+) | `run_scan_now`, `acknowledge_alert`, `mute_site`, `unmute_site` |
| **2 · high impact** | Confirm required (analyst+) | `add_site`, `rebaseline_site`, `set_flag_threshold`, `set_scan_interval` |
| **3 · destructive** | Confirm required (analyst+) | `delete_site` |

<Note>
  Acknowledging an alert or running a scan-now is a **safe** action and executes without a confirmation card. Only changing configuration (thresholds, intervals), adding or rebaselining a site, or deleting a site pauses for confirmation.
</Note>

Tool results are compact JSON — ids truncated, no raw HTML or evidence blobs — for token efficiency and prompt-injection containment. Tool output is treated as *data*, never as instructions.

## The action confirmation loop

Because a model can misinterpret a request, all tier ≥ 2 actions go through a strict confirmation loop. The model never executes a state-changing action directly.

<Steps>
  <Step title="Model proposal">
    The user asks for a high-impact action ("delete the staging site" or "set example.com's flag threshold to 0.8"). The model calls the corresponding tool.
  </Step>
  <Step title="Frozen pending state">
    The dispatcher intercepts the call, freezes the tool name and arguments verbatim into an `agent_pending_actions` row, and surfaces a confirmation card. Only one pending action per conversation exists at a time — proposing a new one cancels the prior.
  </Step>
  <Step title="Explicit user confirmation">
    The UI renders a confirmation card (or Telegram sends a prompt). The user presses Confirm or Cancel — a button, never model output. Cards expire after **10 minutes**.
  </Step>
  <Step title="Re-checked execution">
    On confirm, ownership (the confirmer must be the proposer), the pending status, expiry, and the user's current role are all re-checked. Only then are the frozen arguments executed exactly as stored. A role change or expiry between propose and confirm fails closed.
  </Step>
</Steps>

<Warning>
  **Safety guarantee.** The LLM is never in the loop between propose and execute. It cannot bypass the confirmation, alter the frozen arguments after approval, or escalate its own privileges. The action is marked confirmed *before* execution so a crash can never leave a re-runnable pending row.
</Warning>
