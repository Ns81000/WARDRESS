Wardress is API-first: everything the dashboard does runs over the same REST API. The interactive OpenAPI documentation (Swagger UI) is served locally at **`http://localhost:8321/docs`**, and the raw schema at `/openapi.json`.

<Info>
  All API routes are mounted under the `/api` prefix. The SPA is served from the same origin, so browser sessions and API-key scripts hit identical endpoints.
</Info>

## Authentication

Two credential kinds arrive in the same `Authorization: Bearer` header and are routed by prefix:

- **JWT access tokens** — issued to the dashboard on login, kept in memory, refreshed via an `HttpOnly` cookie.
- **API keys** (`wk_...`) — for scripting. Looked up by SHA-256 hash and carrying the owning user's role, so RBAC applies identically to both.

<Steps>
  <Step title="Generate an API key">
    In the dashboard, go to **Settings → API Keys** and create a key. The raw key is shown **exactly once** — store it securely. It looks like `wk_` followed by 43 URL-safe characters.
  </Step>
  <Step title="Send it as a Bearer token">
    ```bash
    curl -H "Authorization: Bearer wk_your_api_key_here" \
      http://localhost:8321/api/sites
    ```
  </Step>
</Steps>

<Warning>
  **API keys cannot manage credentials.** The endpoints that mint or manage credentials (`/api/api-keys`, `/api/auth/logout`) require an interactive JWT session and reject API keys — a leaked key must not be able to create more credentials. Everything else accepts either credential kind, subject to the key owner's role.
</Warning>

## Roles at the API boundary

RBAC is enforced per endpoint via dependency factories, so a `403` (valid credential, insufficient role) is distinct from a `401` (no/invalid credential).

| Role | Scope |
| :--- | :--- |
| **admin** | Everything, including users, settings, notification channels, remediation hooks, and the audit log. |
| **analyst** | Sites, scans, rebaselines, suppression rules, alert acks, explains, reports, bulk import, and remediation confirm/dismiss. |
| **viewer** | Read-only GET surfaces. |

## Example: managing sites

<CodeGroup>

```bash List sites
curl -X GET "http://localhost:8321/api/sites" \
  -H "Authorization: Bearer wk_your_api_key_here"
```

```bash Trigger a manual scan
curl -X POST "http://localhost:8321/api/sites/{site_id}/scan-now" \
  -H "Authorization: Bearer wk_your_api_key_here"
```

```bash Acknowledge an alert
curl -X POST "http://localhost:8321/api/alerts/{alert_id}/ack" \
  -H "Authorization: Bearer wk_your_api_key_here"
```

</CodeGroup>

## Request limits and rate limiting

- A request-body size limit is enforced by middleware (`413 Request body too large` when exceeded).
- A **per-IP** rate limit runs in middleware before authentication; a **per-user** limit runs in the auth dependency. Both return `429` with a `Retry-After` header. Tune via `RATE_LIMIT_PER_IP`, `RATE_LIMIT_PER_USER`, and `RATE_LIMIT_WINDOW_SECONDS` (see [Configuration](/configuration)).
- CORS is locked to explicitly configured origins (`CORS_ALLOWED_ORIGINS`); by default the SPA is same-origin and no cross-origin request is permitted.

## Hashed API keys

API keys are stored only as SHA-256 hashes. A database compromise cannot recover usable tokens. Lost a key? Revoke it and generate a new one — there is no recovery of the raw value.
